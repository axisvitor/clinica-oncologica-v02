**File**: `scripts/lint_migrations.py`
```python
import re
import sys
from pathlib import Path

REQUIRED_PATTERN = r'^\d{8}_\d{6}_[a-z0-9_]+\.py$'

def lint_migration_name(filepath: Path) -> bool:
    """Ensure migration follows YYYYMMDD_HHMMSS_description.py pattern"""
    filename = filepath.name

    # Allow merge heads and special cases
    if filename.startswith(('merge_', 'baseline_')):
        return True

    if not re.match(REQUIRED_PATTERN, filename):
        print(f"❌ {filename} - Does not match pattern YYYYMMDD_HHMMSS_description.py")
        return False

    print(f"✅ {filename}")
    return True

if __name__ == "__main__":
    migrations_dir = Path("alembic/versions")
    all_valid = all(
        lint_migration_name(f)
        for f in migrations_dir.glob("*.py")
        if f.name != "__init__.py"
    )
    sys.exit(0 if all_valid else 1)
```

## 7. Future Migration Best Practices

### 7.1 Mandatory Standards (ENFORCE)

#### **Standard 1: File Naming**
```
FORMAT: YYYYMMDD_HHMMSS_descriptive_action.py

EXAMPLES:
✅ 20251010_143000_add_patient_consent_tracking.py
✅ 20251011_090000_create_medication_reminders_table.py
✅ 20251012_160000_add_timezone_support_to_appointments.py

❌ add_patient_consent.py
❌ 040_medication_table.py
❌ a3f89c2d1b4e_create_table.py
```

#### **Standard 2: Migration Content Structure**
```python
"""Brief description (1 line)

Revision ID: YYYYMMDD_HHMMSS
Revises: previous_revision
Create Date: YYYY-MM-DD HH:MM:SS

Detailed description:
- What changes are being made
- Why these changes are necessary
- What tables/columns/indexes are affected
- Expected performance impact
- Rollback considerations

Related Issues: #123, #456
"""

from alembic import op
import sqlalchemy as sa

# revision identifiers, used by Alembic
revision = 'YYYYMMDD_HHMMSS'
down_revision = 'previous_revision'
branch_labels = None
depends_on = None


def upgrade() -> None:
    """
    Apply migration changes with detailed comments.

    Each operation should have a comment explaining:
    - What is being created/modified
    - Why this change is needed
    - Expected impact on queries
    """
    # Create index for frequently queried column
    # Improves patient lookup by consent_date by ~50x
    op.create_index(
        'idx_patient_consent_date',
        'patient_consents',
        ['patient_id', 'consent_date'],
        postgresql_using='btree',
        postgresql_concurrently=True,  # Avoid table locks
    )


def downgrade() -> None:
    """
    Rollback migration changes.

    IMPORTANT: Test rollback in development before deploying!
    """
    op.drop_index('idx_patient_consent_date', 'patient_consents')
```

#### **Standard 3: Index Creation Safety**
```python
# ✅ ALWAYS use CONCURRENTLY for production databases
op.execute("""
    CREATE INDEX CONCURRENTLY IF NOT EXISTS idx_name
    ON table_name (column_name)
""")

# ❌ NEVER create indexes without CONCURRENTLY in production
op.create_index('idx_name', 'table_name', ['column_name'])
```

#### **Standard 4: Enum Type Management**
```python
# ✅ SAFE enum modification with checkfirst
status_enum = postgresql.ENUM(
    'active', 'pending', 'cancelled', 'completed',
    name='appointment_status',
    create_type=True
)
status_enum.create(op.get_bind(), checkfirst=True)

# ❌ DANGEROUS - fails if enum exists
op.execute("CREATE TYPE appointment_status AS ENUM (...)")
```

### 7.2 Pre-Deployment Checklist

Before applying any migration to production:

- [ ] Migration tested locally with production-like data volume
- [ ] Downgrade function tested and verified
- [ ] Migration runs with CONCURRENTLY for all indexes
- [ ] Estimated execution time documented (< 5 minutes preferred)
- [ ] Database backup taken immediately before
- [ ] Rollback plan documented and tested
- [ ] No `--sql` mode errors when generating SQL
- [ ] Peer reviewed by at least one other developer
- [ ] Production migration state documented before and after
