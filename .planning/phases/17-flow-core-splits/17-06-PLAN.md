<objective>
Close the final Phase 17 verification gap (truth #13: full backend test suite passes) by fixing the `treatment_phase` response validation mismatch that causes `ResponseValidationError` on `GET /api/v2/patients/`.

**Root cause:** The `PatientV2Base` schema defines `treatment_phase` with a strict regex pattern that only allows 12 specific values (`initial`, `adjustment`, `maintenance`, `monitoring`, `followup`, `completed` and their Portuguese equivalents). However, the database column `patients.treatment_phase` is a free-form `VARCHAR(100)` that already contains values like `'onboarding'` (set by the saga onboarding flow and test fixtures). When the list endpoint serializes patients, FastAPI's response model validation rejects the non-matching value, raising a `ResponseValidationError` that surfaces as HTTP 500.

**Fix strategy:**
1. The pattern constraint is appropriate for **input** (create/update) but must be lenient on **output** (response). Split the `treatment_phase` field so that `PatientV2Response` overrides it without a regex pattern, accepting any persisted value.
2. Add `'onboarding'` to the valid input patterns in both the v2 and domain schemas, since the saga flow legitimately writes this value.

Purpose: Make the patient list endpoint tolerate all persisted `treatment_phase` values without 500 errors, closing the last verification gap for Phase 17.
Output: Patched schemas and full-suite green evidence.
</objective>
