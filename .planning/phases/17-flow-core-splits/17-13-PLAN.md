<objective>
Close the final Phase 17 verification gap (truth #13: full backend fail-fast test gate passes) by fixing the DLQ test fixtures in `tests/api/v2/test_admin_extensions.py` that create `FailedMessage` objects with `patient_id=uuid4()` — a random UUID not matching any patient row in the DB, violating the `whatsapp_delivery_failures_patient_id_fkey` foreign key constraint.

**Root cause (confirmed from 17-12 fail-fast rerun):** The `dlq_items` fixture at lines 80-105 sets `patient_id=uuid4()`. The `whatsapp_delivery_failures` table has `patient_id REFERENCES patients(id) ON DELETE CASCADE` with `nullable=False`. Since no patient with that UUID exists, the INSERT violates the FK constraint. The same pattern recurs in `test_purge_dlq_items_actual` at lines 470-472 which also creates a `FailedMessage` with `patient_id=uuid4()`.

**Fix strategy:**
1. Add a `test_patient` fixture that creates a real `Patient` row using `admin_user.id` as `doctor_id` and a simple name
2. Make the `dlq_items` fixture depend on `test_patient` and use `test_patient.id` as `patient_id`
3. Fix `test_purge_dlq_items_actual` to also use `test_patient.id` instead of `uuid4()` for the old_item

This is a fixture-only change in a single test file. No production code, migrations, models, or constraint guards need modification.

Purpose: Achieve the required fully-green fail-fast gate so Phase 17 can be verified as complete (13/13 truths).
Output: Fixed fixtures + fully-green fail-fast evidence.
</objective>
