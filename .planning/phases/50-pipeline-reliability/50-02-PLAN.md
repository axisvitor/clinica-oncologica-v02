<objective>
Add automatic Celery-based retry with exponential backoff for failed outbound WhatsApp flow messages.

Purpose: Currently, when `_send_flow_message` in `sequencing.py` calls `whatsapp_service.send_message()` and it fails, the method just returns `False`, causing the flow to return `status: "error"`. The patient's flow stalls with no retry. This plan creates a dedicated Celery task `retry_failed_flow_send` that re-attempts delivery with exponential backoff (max 3 attempts), and modifies `_send_flow_message` to enqueue this task on failure instead of silently giving up.

Output: New Celery task `send_retry.py`, modified `sequencing.py`, and tests.
</objective>

The task body should:
1. Open a scoped DB session via `get_scoped_session()`
2. Load Message by id; return early if not found
3. If message status is already SENT/DELIVERED/READ, return early (idempotent)
4. Reset message status to PENDING if it is FAILED
5. Create UnifiedWhatsAppService with db session
6. Call `async_to_sync(service.send_message)(message, flow_context=flow_context)` in try/except
7. On success: return `{"status": "ok", "message_id": message_id, "attempt": self.request.retries + 1}`
8. On ExternalServiceError: calculate countdown = SEND_RETRY_BASE_DELAY * (SEND_RETRY_BACKOFF_FACTOR ** self.request.retries), add jitter (random 0-10s), then `self.retry(countdown=countdown, exc=exc)`
9. On MaxRetriesExceededError: set message.status = MessageStatus.FAILED, commit, log structured error, return `{"status": "permanently_failed", "message_id": message_id, "attempts": SEND_RETRY_MAX_RETRIES}`

Create tests that mock `get_scoped_session`, `UnifiedWhatsAppService.send_message`, and verify:
- Success path returns ok
- ExternalServiceError triggers self.retry with correct countdown
- MaxRetriesExceededError marks message as FAILED
- Missing message returns early
- Idempotent skip for already-sent messages

1. After `success = await self.whatsapp_service.send_message(message, flow_context=flow_context)`:
   - If `success` is False (send failed but no exception), enqueue retry:
     ```python
     from app.tasks.flows.send_retry import retry_failed_flow_send
     retry_failed_flow_send.apply_async(
         args=[str(message.id)],
         kwargs={"flow_context": flow_context},
         countdown=60,  # first retry after 60s
     )
     logger.warning(
         "Flow message send failed, enqueued retry",
         extra={
             "message_id": str(message.id),
             "patient_id": str(patient.id),
             "flow_kind": flow_kind,
             "day_number": day_number,
             "message_index": message_index,
         },
     )
     ```
   - Still return False so the immediate caller knows send hasn't completed yet.
   - But now the retry task will handle eventual delivery.

2. In the existing exception handling block (`except Exception` in the callers like `_send_all_sequential`, `_send_wait_each_with_auto_advance`, etc.), the "error" status is still returned to the flow, but the retry task will independently attempt re-delivery.

3. Add the import at the top of the file (lazy import inside the if-block to avoid circular imports):
   ```python
   # Inside the if not success block:
   from app.tasks.flows.send_retry import retry_failed_flow_send
   ```

This is a minimal, surgical change. The flow still returns error status to the caller, but the retry task provides a background safety net for eventual delivery.
