       ```python
       async def _adk_sentiment(response: str, patient_context: str = "{}") -> str:
           """Analyze patient response sentiment."""
           # This will be called by ADK Runner, which passes string args
           # Actual execution delegates to the real handler via the runtime context
           return json.dumps({"tool": "sentiment", "response": response, "patient_context": patient_context})
       ```
       NOTE: These ADK-facing wrappers are schema stubs for the Runner's tool dispatch. The actual domain logic still flows through the handler functions when the Runner calls these tools. The Runner calls FunctionTool -> ADK extracts the tool call -> runtime catches the tool invocation and routes to the real handler with proper deps/context.
    3. ALTERNATIVE SIMPLER APPROACH (preferred): Since the ADK Runner+Agent pattern is designed for LLM-driven tool selection, but our use case is deterministic (user specifies tool_name), use `FunctionTool` wrapping for ADK-06 compliance while keeping dispatch deterministic:
       - Create `FunctionTool(func=sentiment_tool_adk_compat)` where `sentiment_tool_adk_compat` has ADK-friendly signature.
       - Expose `get_adk_function_tools()` returning `dict[str, FunctionTool]` alongside existing `get_tool_registry()`.
       - Use `try/except ModuleNotFoundError` for `FunctionTool` import; when unavailable, return lightweight dataclass wrappers that store the function reference.
    4. Keep `get_tool_registry()` as-is (backward compat for existing callers).

Iterate events to extract final response text.
