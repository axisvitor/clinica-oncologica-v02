From orchestrator.py constructor:
```python
def __init__(self, db: Any, redis_client=None, evolution_client=None):
    self.db = db  # Could be AsyncSession OR sync Session
    self.step_executor = SagaStepExecutor(db=db, ...)  # AsyncSession-native
    self.compensator = SagaCompensator(db=db, ...)      # AsyncSession-native
    self.persistence = SagaPersistence(db)               # sync Session-native
```

Key finding: orchestrator.py has ~15 sync DB calls (self.db.query(), self.db.commit(),
self.db.rollback(), self.db.add(), self.db.flush()) inside async methods. When called
from the API path with AsyncSession, these WILL fail at runtime because AsyncSession
does not support sync .query() or sync .commit().

The Celery path wraps async calls with run_async(), meaning the saga orchestrator receives
a sync Session from Celery but the step_executor/compensator expect async patterns.
This works because run_async() creates an event loop for the async methods, and the
sync Session objects can be awaited if the methods don't actually use await on them.

FIX STRATEGY: Convert orchestrator.py's direct DB calls from sync to async (using
`await self.db.execute(select(...))` instead of `self.db.query(...)`, and
`await self.db.commit()` instead of `self.db.commit()`). The Celery path will continue
to work because run_async() already runs the async method in an event loop, and
SQLAlchemy's AsyncSession bridge handles both callers.
