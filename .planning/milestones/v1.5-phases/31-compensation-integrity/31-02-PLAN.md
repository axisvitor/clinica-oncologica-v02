# Plan 31-02: Verify DB Transaction Boundaries and Idempotency Guards

## Goal

Add verification tests to the compensation integrity test module proving that (1) forward saga steps use flush-only within a single uncommitted transaction that rolls back atomically on failure (COMP-03), and (2) both forward steps and compensation handlers are idempotent under re-execution (COMP-04). These tests use mocks to verify the DB call contract rather than requiring a real database.

## Must-Haves (goal-backward)

- [ ] A test proves forward steps call `_db_flush()` (not commit) and the orchestrator issues exactly one `_db_commit()` at the end of the forward path, confirming the Unit-of-Work pattern
- [ ] A test proves that on forward failure, `_db_rollback()` is called before any failure record is persisted, confirming no partial writes leak
- [ ] A test proves the compensation path issues its own `_db_commit()` independent of the forward transaction
- [ ] A test proves each compensation handler's idempotency guard (`compensated_steps` check) prevents double-execution when the handler name is already in the list
- [ ] A test proves forward step idempotency: `step_initialize_flow` with an existing `PatientFlowState` skips creation; `step_send_welcome_message` with an existing message skips send
- [ ] All tests pass: `cd backend-hormonia && python -m pytest tests/unit/orchestration/test_saga_compensation_integrity.py -x -q`

## Interfaces

From `backend-hormonia/app/orchestration/saga_orchestrator/orchestrator.py` — forward transaction pattern:
```python
# Inside execute_patient_onboarding_saga():
try:
    patient = await self.step_executor.step_create_patient(...)    # flush only
    await self.step_executor.step_initialize_flow(...)              # flush only
    await self.step_executor.step_send_welcome_message(...)         # flush only
    saga.status = SagaStatus.COMPLETED
    await self._db_commit()   # SINGLE commit for all forward work
except Exception as e:
    await self._db_rollback()  # ALL forward flushes discarded
    # ... create failure_saga record in new transaction ...
    self.db.add(failure_saga)
    await self._db_commit()    # Second tx: persist FAILED record only
    await self.compensator.compensate_saga(failure_saga)
```

From `backend-hormonia/app/orchestration/saga_orchestrator/db_adapter.py`:
```python
class SagaDBAdapterMixin:
    async def _db_flush(self) -> None: ...
    async def _db_commit(self) -> None: ...
    async def _db_rollback(self) -> None: ...
```

From `backend-hormonia/app/orchestration/saga_orchestrator/compensation.py` — compensation transaction:
```python
# Inside _compensate_saga_internal():
saga.status = SagaStatus.COMPENSATING
# ... run handlers ...
saga.status = SagaStatus.COMPENSATED
await self._db_commit()   # Compensation's own commit
```

Idempotency guard in all three handlers (`compensation_handlers.py`):
```python
compensated_steps = saga.step_data.get("compensated_steps", []) if saga.step_data else []
if "message" in compensated_steps:  # or "flow" or "patient"
    return  # skip
```

```python
# step_send_welcome_message: queries for existing Message with matching idempotency_key, returns early if found
```

<title>Create COMP-03 transaction boundary verification tests</title>
<context>
COMP-03 requires proving that forward saga steps execute within flush-only boundaries (no individual commits), with one atomic commit at the end. On failure, rollback discards all forward flushes. Compensation runs in its own transaction. The research identified the two-transaction model: forward (flush+commit or flush+rollback) and compensation (separate commit).

The test approach: read the source code of orchestrator.py and steps.py to statically verify the flush-not-commit pattern in forward steps, then use mocks to verify the orchestrator's commit/rollback call sequence.
</context>

Add to `backend-hormonia/tests/unit/orchestration/test_saga_compensation_integrity.py`:

<title>Create COMP-04 idempotency guard verification tests</title>
<context>
COMP-04 requires proving that re-executing any saga step or compensation handler produces the same final state. Compensation handlers use a `compensated_steps` list in `saga.step_data` to skip already-compensated steps. Forward steps check for existing records (Patient by IntegrityError, PatientFlowState by query, Message by idempotency_key query) to avoid duplicates.

The research also confirmed that even without the in-memory guard, the DB-level operations are inherently idempotent: cancelling an already-cancelled message is a no-op, deleting a non-existent flow state is a no-op, deleting a non-existent patient is a no-op.
</context>
<action>
Add to `backend-hormonia/tests/unit/orchestration/test_saga_compensation_integrity.py`:

**Class: `TestIdempotencyGuards` (COMP-04)**

Fixtures: `mock_db` (MagicMock, with query returning appropriate mock chains), `make_saga` factory.

**Compensation idempotency tests (mock-based, async):**

1. `test_compensate_message_skips_when_already_compensated` — Create saga with `step_data={"compensated_steps": ["message"]}`. Call `await compensate_message(mock_db, saga)`. Assert `mock_db.query` was NOT called (the handler returns early before querying). Import `compensate_message` from `app.orchestration.saga_orchestrator.compensation_handlers`.

2. `test_compensate_flow_skips_when_already_compensated` — Create saga with `step_data={"compensated_steps": ["flow"]}`. Call `await compensate_flow(mock_db, saga)`. Assert no DB query. Import `compensate_flow`.

3. `test_compensate_patient_skips_when_already_compensated` — Create saga with `step_data={"compensated_steps": ["patient"]}`. Call `await compensate_patient(mock_db, saga)`. Assert no DB query. Import `compensate_patient`.
