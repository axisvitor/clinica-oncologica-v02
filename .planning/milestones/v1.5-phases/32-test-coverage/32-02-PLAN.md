# Plan 32-02: Compensation Rollback Test Suite (Per-Step Handler Verification)

## Goal

Create a behavioral test suite that exercises each of the three compensation handlers (`compensate_patient`, `compensate_flow`, `compensate_message`) against mocked DB state and verifies they produce the correct cleanup actions. Unlike Phase 31 tests (which assert idempotency guards via mock call counts), these tests set up realistic pre-conditions (records in mock DB matching what the forward step would create) and verify the handler mutates them correctly: patient hard-deleted, flow record deleted, message marked CANCELLED. Additionally, test that calling `SagaCompensator.compensate_saga()` after a step-4 failure runs all three handlers and leaves the saga in COMPENSATED status.

## Must-Haves (goal-backward)

Tracked in YAML frontmatter. Checklist for executor reference:
- [ ] A test exercises `compensate_patient(db, saga)` with a mock patient record and verifies `db.delete(patient)` is called (hard-delete)
- [ ] A test exercises `compensate_flow(db, saga)` with a mock PatientFlowState and verifies the DB delete was issued for that flow record
- [ ] A test exercises `compensate_message(db, saga)` with a mock pending Message and verifies `message.status == MessageStatus.CANCELLED`
- [ ] A test exercises the full compensation sequence via `SagaCompensator._compensate_saga_internal` at current_step=4 and verifies all three handlers run and saga.status becomes COMPENSATED
- [ ] Each handler test verifies the handler appends its step name to `saga.step_data["compensated_steps"]` (no orphan state)
- [ ] All tests pass: `cd backend-hormonia && python -m pytest tests/unit/orchestration/test_saga_compensation_exercise.py -x -q`

TEST-02 requires exercising each compensation handler and verifying correct cleanup. The three handlers are:
1. `compensate_patient` — hard-deletes patient via `db.delete(patient)` (patient was never fully onboarded)

**Project fixtures:** The `mock_redis` fixture from `tests/fixtures/saga_fixtures.py` can be used for the SagaCompensator instantiation. The `mock_failing_compensation` and `mock_successful_compensation` fixtures are available but not needed here since we test real handlers against mock DB state, not mock handlers. The `make_saga` factory defined inline is intentional because it needs precise control over `step_data` dict contents (empty vs pre-populated compensated_steps), which the project's `failed_saga_record` fixture does not support (it creates a DB-persisted saga with fixed state).
