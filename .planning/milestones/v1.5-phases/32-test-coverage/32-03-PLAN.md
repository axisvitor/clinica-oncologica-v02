# Plan 32-03: Edge Case Tests and Flow State Lifecycle Tests

## Goal

Create two test modules: (1) saga edge-case tests covering timeout handling, concurrent saga execution guard, and step retry exhaustion (TEST-03), and (2) flow state lifecycle tests covering pause-mid-saga, resume-after-pause, and cancel-during-execution (TEST-05). These fill the final gaps identified in the Phase 32 research where no behavioral tests exist for these failure modes and state transitions.

## Must-Haves (goal-backward)

Tracked in YAML frontmatter. Checklist for executor reference:
- [ ] A test proves that saga execution under timeout conditions sets saga status to FAILED with an appropriate error_message (not COMPLETED)
- [ ] A test proves that concurrent saga execution for the same patient phone is guarded by the distributed lock — a second call while locked raises or returns an error result
- [ ] A test proves that step retry exhaustion (max_retries exceeded) triggers compensation rather than leaving the saga in a partial state
- [ ] A test exercises pause-mid-flow: an active flow transitions to `paused` status with `state_data["paused"] = True`
- [ ] A test exercises resume-after-pause: a paused flow transitions back to `active` status with `state_data["paused"] = False`
- [ ] A test exercises cancel-during-execution: an active flow transitions to `cancelled`, pending messages are marked CANCELLED, and saga compensation is NOT triggered (independent lifecycles)
- [ ] All tests pass: `cd backend-hormonia && python -m pytest tests/unit/orchestration/test_saga_edge_cases.py tests/unit/services/test_flow_lifecycle.py -x -q`

<context>
TEST-03 requires edge-case tests for saga timeout, concurrent execution, and step retry exhaustion. The research found:
1. The orchestrator wraps execution in `asyncio.wait_for(coro, timeout=self.global_timeout)` — timeout triggers except block and compensation.
2. Concurrent execution is guarded by `acquire_lock(f"saga:onboarding:{phone_hash}")` — if the lock is already held, a `LockError` or similar exception is raised.
3. Step retry uses `_compensate_step_with_retry` with `max_retries=3` — exhaustion logs the error and continues compensation (does not re-raise).

The existing integration test `test_saga_timeout_handling` in `test_patient_saga.py` only tests manual status setting; it does not exercise the timeout machinery.

**Project fixtures:** Use `mock_redis` and `mock_evolution_client` from `tests/fixtures/saga_fixtures.py` for Redis/Evolution mocking. The `_noop_acquire_lock` pattern comes from existing `tests/orchestration/test_saga_orchestrator.py`. The `_blocking_acquire_lock` helper is defined inline since no project fixture exists for lock-failure simulation. The `mock_failing_action` fixture from saga_fixtures.py can be used for step failure simulation in retry exhaustion tests.
</context>
