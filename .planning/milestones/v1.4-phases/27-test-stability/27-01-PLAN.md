**auth_session_shared.py — Fix db.query(User) in async callback**

The `_fetch_user_by_uid` nested function at line 64-65 calls `db.query(User).filter(...).first()`
which raises MissingGreenlet when `db` is an AsyncSession (passed from `localization.py` and
`tasks/dependencies.py` which both inject `Depends(get_async_db)`).

Replace the nested function to use an awaitable-resolution pattern that works with both Session
and AsyncSession:

```python
async def _fetch_user_by_uid(uid: str):
    from sqlalchemy import select
    from app.models.user import User
    stmt = select(User).where(User.firebase_uid == uid)
    result = db.execute(stmt)
    # If db is AsyncSession, execute() returns a coroutine that must be awaited.
    # If db is sync Session, execute() returns the result directly.
    import inspect
    if inspect.isawaitable(result):
        result = await result
    return result.scalar_one_or_none()
```

This pattern (already used extensively in Phases 22-26) resolves both Session and AsyncSession
without branching in the caller. The `inspect.isawaitable` check is the standard dual-mode
resolution helper used throughout the codebase.

Do NOT change the function signature of `get_user_data_from_session` — all callers remain stable.
Do NOT change `user_cache_shared.py` — it correctly awaits the callback already.

Add the `select` import at the top of `auth_session_shared.py`:
```python

```

**Problem:** `test_bulk_delete_success` fails with HTTP 400 because the `get_admin_user` test-mode
fallback at `app/api/v2/routers/admin/dependencies.py:72-76` queries for ANY active admin user
in the DB. When `multiple_users` fixture creates additional active admins (odd indices), two
failure modes exist:
1. `scalar_one_or_none()` raises `MultipleResultsFound` (500) if multiple active admins exist
2. If only one extra admin happens to match, it could be in the delete list (self-delete → 400)

**Fix:** Add an explicit `get_admin_user` dependency override in the test that returns the
`admin_user` fixture directly, bypassing the DB fallback. This ensures the authenticated admin
is always the fixture-created admin, never a `multiple_users` entry.

In `tests/api/v2/test_admin.py`, add an `autouse=True` fixture inside `TestBulkDelete` that
overrides the admin dependency:

```python
class TestBulkDelete:
    """Tests for POST /api/v2/admin/users/bulk-delete"""

    @pytest.fixture(autouse=True)
    def _override_admin_dep(self, admin_user):
        from app.main import app
        from app.api.v2.routers.admin.dependencies import get_admin_user

        async def _admin_override():
            return admin_user

        app.dependency_overrides[get_admin_user] = _admin_override
        yield
        app.dependency_overrides.pop(get_admin_user, None)
```

Also ensure `multiple_users[:3]` does NOT include the `admin_user` fixture. The current fixture
creates users with email `user0@test.com` through `user24@test.com` while `admin_user` has email
`admin@test.com`, so there is no ID collision — the override alone fixes the issue.

Do NOT change other test classes. Only add the autouse fixture inside `TestBulkDelete`.
